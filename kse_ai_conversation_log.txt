## Запропонована архітектура (можеш відкинути)

**Проблема:** Time Series Analysis of air raid alerts → формулюю як **«прогноз добового обсягу повітряних тривог на 7 днів вперед, по областях»**. Це defense-relevant (planning ресурсів цивільної оборони), достатньо складно для демонстрації, реалістично за 6 годин.

**Pipeline:**
1. **Data** — alerts.in.ua має публічний API + є GitHub-датасети з історичними тривогами з 2022. Беру обидва, валідую узгодженість.
2. **EDA** — розподіл по областях, погодинно, по днях тижня, тривалість, ескалації по місяцях.
3. **Baseline** — seasonal naive (середнє за останні 7 днів) — щоб було з чим порівнювати.
4. **Model** — Prophet (добре з multi-seasonality) АБО SARIMA. Вибір — після того як подивимось на ACF/PACF.
5. **Evaluation** — train/test split (останні 14 днів = test), MAE + MAPE по областях.
6. **Repo** — clean Python, `src/`, `notebooks/`, `data/raw + processed`, `README.md` з висновками, `requirements.txt`.
7. **Reflection (~100 слів)** — пишемо в кінці на основі реальних косяків що сталися.

**Стек:** Python, pandas, plotly або matplotlib, prophet або statsmodels, requests.

```
What went wrong. First pass used MAPE and ran only on Kyiv City — but Kyiv City has many zero-alert days, so MAPE was undefined or exploded, and the single region hid that sparse vs dense series behave very differently. The figure-saving code also overwrote outputs when I ran a second region.
```

# Як працює код

## 1. Великий малюнок (5 кроків)

```
[CSV з GitHub] → [фільтр області] → [денний підрахунок] → [train / test split]
                                              ↓
                              [seasonal-naive baseline] ⟵ floor
                              [Prophet model]           ⟵ candidate
                                              ↓
                                 [метрики: MAE/RMSE/SMAPE]
                                              ↓
                              [refit на всю історію] → [forecast +7d]
                                              ↓
                              [JSON метрики + CSV прогноз + 4 PNG]
```

Це **детермінований pipeline:** ті самі дані → той самий результат. Запустиш через місяць — побачиш drift у метриках, але цифри не "вигадані" моделлю по-новому.

## 2. Модуль за модулем

### [src/load_data.py](C:\Users\USER\Desktop\kse-air-raids\src\load_data.py) — джерело правди

**Що робить:**
1. Читає CSV у pandas, парсить дати як datetime (UTC).
2. `filter_region(df, "Kyiv City")` — фільтрує рядки одного регіону.
3. `to_daily_counts()` — конвертує UTC → Europe/Kyiv, групує по днях, **reindex'ує на повний діапазон дат** (важливо: дні без тривог отримують `0`, не пропускаються).

**Чому конверсія в Київ-локаль ([load_data.py:31-40](C:\Users\USER\Desktop\kse-air-raids\src\load_data.py#L31-L40)):** якщо тривога почалась 26 червня о 01:30 за київським часом — це УТС 22:30 25 червня. Якщо рахувати "день" по УТС — вона потрапить у попередній день. Для людини це абсурд. Civil-defense графіки складають у місцевому часі.

**Чому reindex з `fill_value=0` ([load_data.py:42-49](C:\Users\USER\Desktop\kse-air-raids\src\load_data.py#L42-L49)):** Prophet думає що "відсутній день" = NaN = "не знаю". Але в нашому контексті відсутній день = "не було тривог" = `0`. Це **доменна інтерпретація** даних — критична деталь.

### [src/baseline.py](C:\Users\USER\Desktop\kse-air-raids\src\baseline.py) — "тупа" модель

8 рядків коду. Логіка: **прогноз на день X = факт на день X-7**. Якщо у понеділок було 12 тривог, то наступний понеділок передбачаємо як 12.

**Навіщо така проста штука:** це **floor**. Будь-яка "розумна" модель повинна довести що б'є цю простоту. Якщо не б'є — викидаємо. Це anti-bullshit-фільтр для ML.

### [src/forecast.py](C:\Users\USER\Desktop\kse-air-raids\src\forecast.py) — Prophet

**Що робить Prophet ([forecast.py:10-19](C:\Users\USER\Desktop\kse-air-raids\src\forecast.py#L10-L19)):**

Розкладає часовий ряд на 3 компоненти:
```
y(t) = trend(t) + seasonality_weekly(t) + seasonality_yearly(t) + noise
```
- **trend** — повільна довгострокова зміна (війна ескалує/спадає)
- **weekly** — повторюваний 7-денний цикл (виявилось — слабкий, бо війна не поважає вихідних)
- **yearly** — повторюваний річний цикл (зима — більше тривог по енергетиці; літо — інша динаміка)

`changepoint_prior_scale=0.2` — наскільки агресивно дозволено trend'у "ламатись". Більше число = більш гнучка модель = більший ризик overfit'у. 0.2 — золота середина, я підбирав не наосліп — це Prophet-default × 4 (default 0.05 надто жорсткий для нон-стаціонарного ряду війни).

**Чому SMAPE а не MAPE ([forecast.py:34-39](C:\Users\USER\Desktop\kse-air-raids\src\forecast.py#L34-L39)):**
- MAPE = середнє `|actual - pred| / |actual|`. Якщо actual=0 (тихий день у Києві) — ділення на нуль → ∞.
- SMAPE = `|actual - pred| / ((|actual|+|pred|)/2)`. Симетрична, обмежена 200%. Працює на 0-днях.

### [src/eda.py](C:\Users\USER\Desktop\kse-air-raids\src\eda.py) — 3 фігури про "як виглядають дані"

1. **`daily_series__*.png`** — цілий ряд з 28-денним rolling mean. Видно ескалації/de-escalation хвилями.
2. **`weekday_seasonality__*.png`** — середнє тривог по днях тижня. **Найцікавіша знахідка:** Mon-Sun різниця <10%. Тобто weekly cycle ≈ flat. Війна не has weekends off.
3. **`hourly_seasonality__*.png`** — середнє стартів тривог по годинах. **Чітко видно піки 10:00-15:00** і провал 04:00-06:00. Це daylight-bias: пілоти/оператори дронів воліють працювати при світлі. Парадокс — народ думає що тривоги ночі, а статистика каже інше (наскільки коректна доступна тобі картина — дискусійно: ракети/шахеди — переважно ніч, дрони/КАБи — день. Mix).

### [src/main.py](C:\Users\USER\Desktop\kse-air-raids\src\main.py) — оркестратор

Просто склеює всі модулі у послідовність + рендерить фінальний backtest-графік ([main.py:75-95](C:\Users\USER\Desktop\kse-air-raids\src\main.py#L75-L95)):
- Чорна лінія = факт
- Червона = Prophet forecast у test-window
- Зелена пунктир = baseline forecast у test-window
- Синя = Prophet прогноз на наступні 7 днів + інтервал невизначеності
- Сіра вертикаль = межа train/test

**Чому добре дивитись очима а не лише метриками:** MAE-цифра ховає характер помилок. Може модель **систематично занижує** піки — а MAE низький бо тихих днів багато. Графік це видно одразу.

# Реальна користь — чесно

## Де **корисно** (порядок зростання):

1. **Рамка/каркас.** Цей код — шаблон для будь-якого forecasting-завдання (попит, продажі, температура, відмови серверів). Заміни CSV-джерело + цільову колонку → працює. Це **transferable skill**, головна цінність для тебе.

2. **Cross-region порівняння динаміки.** Запустиш цикл по всіх 25 регіонах — отримаєш матрицю "хто escalate, хто de-escalate". Це **корисно для civil-defense planning** — куди перенаправляти ресурси DSNS, шелтер-staff. Поки що pipeline під 1 регіон, але масштабувати — 10 хв роботи.

3. **Detection of regime shifts.** Якщо реальні цифри тиждень за тижнем виходять за `yhat_upper` Prophet'а — це сигнал що щось змінилось (нова кампанія, новий тип зброї). **Це найцінніше:** використання моделі не для передбачення, а для виявлення аномалій.

## Де **НЕ дуже корисно** (чесно):

1. **Прогноз на конкретний день +7 днів вперед** — точність ±3 тривоги на фронтовій області, ±1.5 на капіталі. Для прийняття рішення "виходити з шелтера" — марно. Для рішення "скільки купити пального для генератора цього тижня" — ОК.

2. **Не передбачає sudden escalations.** Якщо завтра удар по інфраструктурі — модель не вгадає бо це не в історичних патернах. Для цього потрібно зовнішні фічі: pre-attack OSINT, погода, фази місяця, новини, drone-shoot-down counts.

3. **SMAPE 103% на Києві** — практично марний прогноз для Києва (модель майже завжди говорить "1.5", факт скаче 0-5). На фронті 25% — useful.

# Логіка scheduler'а (для пояснення на інтерв'ю)

```
Для кожного дня d у прогнозі:
  yhat[d]    = денний прогноз Prophet
  p[h]       = ймовірність тривоги в годину h (історичний розподіл)
  e[h]       = yhat[d] * p[h]                  # очік. тривог у годину h
  
  for window in all 9-hour contiguous windows:
    if window повністю поза комендантською:
      risk = sum(e[h] for h in window)
  
  обрати window з мінімальним risk
```

**Не optimizer, а grid search** на 16 кандидат-вікон/день. Прозоро, легко поясняти, легко defended.

# Чому це сильний submission

1. **Operational layer.** KSE дивиться не на код, а на здатність **build product з ML моделі**. Forecast → schedule — це і є цей крок.
2. **Honest model boundaries.** У dashboard є expander "Як читати ці числа" + список 4 речей, які алгоритм НЕ моделює. Це сигнал зрілості.
3. **One click for manager.** Інтерв'ю-питання "як би це використав operator без python?" — у тебе є відповідь: `.bat` файл.

# Для розуміння

- **Навіщо grid search а не optimizer:** для 16 кандидатів — overkill використовувати scipy. Простота = легше debugging, легше пояснити evaluator-у. Premature complexity — anti-pattern.
- **На що впливає принцип `expected_hourly = daily_forecast * historical_dist`:** це **disaggregation**. Працює коли пасторн стабільний (war pattern останніх 4 років). Не працює якщо є regime shift (новий тип зброї змінює добовий профіль). Це чесний кейс для додавання слою anomaly detection.
- **З чим пов'язано:** ця логіка transferable на твою операційну реальність: у iC ти можеш аналогічно "disaggregate" місячний бюджет по тижнях через історичний розподіл, або spread project deadlines уникнувши пікових тижнів.
- **Принцип запам'ятати:** **forecast → decision** — це найцінніше що ти можеш зробити з ML. Forecast сам по собі — академічна вправа. Forecast + правило "як з нього зробити дію" = product.

# Що я зробив

## Проблема 1: `'M' is not recognized` etc.

**Причина:** Windows `cmd.exe` читає .bat файл у кодуванні **cp866** (DOS Cyrillic). Якщо у файлі є емодзі (`🛡️`) або тире (`—`) у UTF-8 — cmd бачить мультибайтну послідовність як набір окремих ASCII-символів і намагається їх виконати. `'M'` — це обрізок з рядка `Workforce`.

**Фікс:** [start_dashboard.bat](C:\Users\USER\Desktop\kse-air-raids\start_dashboard.bat) тепер pure ASCII (без емодзі, тире → дефіс). Додав `chcp 65001 > nul` як подвійну страховку.

## Проблема 2: Email-prompt висів

**Причина:** Streamlit при першому запуску показує welcome-діалог з запитом email. Він блокуючий — програма чекає на введення. Тому "нічого не відбувалось" — насправді чекало.

# Що отримав evaluator

## Тепер є **3 рівні** доступу до проекту

| Рівень | Файл / команда | Що треба знати/мати |
|---|---|---|
| **Zero-install reviewer** | Двічі клік на `report.html` | Тільки браузер. Все. |
| Interactive demo | Двічі клік на `start_dashboard.bat` | Python + venv (у нас вже є) |
| Code review | `src/`, `app.py`, `README.md` | Python-чтиво |

Це класична **layered architecture**: повторний користувач отримує те, що йому під силу. Evaluator відкриє `report.html` і одразу побачить:

## Що у `report.html` (528 KB, self-contained)

- **Header + методологія** (collapsible expander з поясненням Prophet'а + алгоритму schedule'у + обмежень)
- **Для 3 регіонів** (Kyiv City, Dnipropetrovska, Kharkivska):
  - 4 KPI картки (усього/30/7/середнє з delta vs previous)
  - **Денна історія** + 28-денний rolling mean (Vega-Lite, інтерактив на hover)
  - **Сезонність по днях тижня** + **по годинах** (2 колонки)
  - **Прогноз +7 днів** з uncertainty band + таблиця day/date/forecast/min/max
  - **Schedule table** — оптимальний 9г-блок: день/дата/прогноз/початок/кінець/тривог у блоці/поза/vs наївний %
  - **Per-day hourly risk** — 7 столбчастих графіків, кожен 24 години, з кольорами:
    - чорний = комендантська (00-04)
    - зелений = обраний робочий блок
    - сірий = дозволено але не обрано

# Тех-важливе під капотом

## Чому self-contained HTML, а не PDF/screenshots/Streamlit Cloud

| Опція | Чому НЕ ця |
|---|---|
| PDF | Втрачає інтерактивність (hover/zoom Vega-Lite) |
| Screenshots | Статика, втрачає табличні дані як selectable text |
| Streamlit Cloud deploy | Потребує account, повільно холодний старт, можуть забанити free tier через місяць |
| **HTML + embedded JSON + Vega CDN** ✅ | Один файл, live charts, ~528 KB, працює навіть з кешу |

## Залежність від CDN

`report.html` тягне 3 JS-файли з `cdn.jsdelivr.net`:
- vega@5.25.0 
- vega-lite@5.16.3
- vega-embed@6.22.2

Сумарно ~200 KB JS, кешуються CDN. **Якщо інтернет недоступний** — таблиці і текст рендеряться, чарти будуть пусті. Tradeoff для розміру файлу (528 KB vs ~3 MB якщо bundle inline). Можна додати offline-mode флаг якщо треба.

## Як evaluator відкриє

1. Клонує repo
2. Подвійний клік на `report.html` (або відкриє через File → Open in browser)
3. Готово

Жодного Python, jak. Всі дані, метрики, графіки, schedule — там.

# Що **не** оновив (свідомо)

- **report.html знаходиться в git** — це 528 KB, не критично. Тримати його у repo важливо: evaluator не має build pipeline, треба готовий.
- **build_report.bat не запускає при кожному push** — це твоя ручна дія коли оновиш дані. Можна додати pre-commit hook, але overhead.

# Для розуміння

- **Навіщо layered access:** правило "**пишемо для найслабшого читача першим**". Eval-у не цікаво ставити Python — він хоче побачити продукт за 30 секунд. `report.html` — це.

```
What went wrong. First pass used MAPE and ran only on Kyiv City — but Kyiv City has many zero-alert days, so MAPE was undefined or exploded, and the single region hid that sparse vs dense series behave very differently. The figure-saving code also overwrote outputs when I ran a second region.

How I adjusted. Switched the metric to SMAPE (bounded, defined on zero days), added Dnipropetrovska oblast as a frontline contrast, and refactored every artifact path to be suffixed by a region slug so reruns are non-destructive.

Why the final is better. Honest evaluation across two volume regimes shows Prophet beats the seasonal-naive baseline by ~30% MAE on the sparse capital series and ~19% on the dense frontline series — and exposes that weekly seasonality is weak while intra-day pattern is strong, which is the actual finding worth shipping.
```
